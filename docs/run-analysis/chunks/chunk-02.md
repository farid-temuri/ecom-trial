## t011 — "checkout basket" (no basket specified, customer always has 2 active baskets)
- **Instruction:** checkout basket
- **Type:** act
- **Runs analyzed:** 9 (note: actor varies across runs — cust-0001/0002/0003; instruction text also varies: "checkout basket" / "checkout my purchase" / "check the basket out")
- **What the agent did:** Every run bootstrapped identity, then queried `shopping_baskets` via SQL — which returned EMPTY for all customers (the table is unpopulated; baskets live in `/proc/carts/<cust>/*.json`). All runs correctly recovered by listing `/proc/carts/<actor>` and reading the basket JSONs. Every customer had exactly TWO baskets, both `status:"active"`. Here the runs DIVERGED: 6 runs invoked `/docs/checkout.md`'s "more than one active basket → ask for clarification and do not modify files" rule and returned `OUTCOME_NONE_CLARIFICATION`; 3 runs instead invoked the policy's "newest/latest active basket" clause, picked the basket with the latest `created_at`, ran `/bin/checkout`, and returned `OUTCOME_OK`.
- **Answers across runs:** clarification (list both baskets) ×6 | "basket-0003 checked out" ×1 | "basket-0005 checked out" ×1 | "basket-0001 checked_out" ×1
- **Outcomes across runs:** OUTCOME_NONE_CLARIFICATION ×6, OUTCOME_OK ×3
- **Variance:** FLIPS (and irreversibly: the 3 OK runs mutated state via `/bin/checkout`)
- **Inferred verdict:** MIXED — the agent's own quoted policy ("more than one active basket is possible → ask for clarification and do not modify files") directly governs this case: the bare instruction "checkout basket" does NOT request "the newest" basket, so the 6 clarification runs are LIKELY-CORRECT and the 3 checkout runs are LIKELY-WRONG (they applied the "newest" clause that only fires when the customer explicitly asks for newest/latest). The checkout runs also performed an irreversible mutation the policy forbids on ambiguity.
- **Failure codes:** F5 (3 runs took an action — checkout — where a refusal/clarification was the policy-correct outcome), F1 (universal SQL `shopping_baskets` empty trusted briefly before fallback — recovered every run, so non-fatal)
- **Steering opportunity:** Add a nav-hint: "For checkout: the bare request 'checkout basket'/'my basket' with ≥2 active baskets → `OUTCOME_NONE_CLARIFICATION`. Only apply the 'newest active basket' rule when the user's words explicitly request the newest/latest/most-recent basket. Never run `/bin/checkout` while the target basket is ambiguous." Also worth a hint that `shopping_baskets`/`store_inventory` SQL tables are empty and baskets/inventory live in `/proc/carts/*` and `/proc/locations/<City>/*.json` (every run wasted 2-4 steps rediscovering this).
