# Chunk-06 — t051–t060

## t051 — "check the basket out" (customer with two active baskets)
- **Instruction:** check the basket out
- **Type:** act
- **Runs analyzed:** 9
- **What the agent did:** Bootstraps identity (actor varies per trial seed: cust-0001, cust-0002, cust-0004). SQL is genuinely DOWN this task (`/bin/sql` returns empty / `Login timeout expired ... PowerTools PROD MS SQL cluster is down`), so every run correctly falls back to `/proc/carts/<cust>/`. Every run finds the actor has **two active baskets**. Runs then diverge hard on interpretation: some treat "check the basket out" as ambiguous "my basket" → ask clarification; others apply the checkout.md "latest active basket" rule → pick the newest `created_at` and run `/bin/checkout`.
- **Answers across runs:** clarification ("two active baskets, which one?") ×4 | checkout completed (basket-0007 / basket-0003) ×2 | no-answer/budget-exhausted ×3 (final answer empty, outcome=undefined)
- **Outcomes across runs:** OUTCOME_NONE_CLARIFICATION ×4, OUTCOME_OK ×2, undefined/ERR ×3
- **Variance:** FLIPS (hard — three distinct terminal behaviors)
- **Inferred verdict:** UNCERTAIN, leaning MIXED — the agent's own retrieved data is consistent (two active baskets every run), so this is a genuine policy-interpretation ambiguity, not a data error. checkout.md is cited by all but contains both a "latest active basket" rule and a "multiple active → clarify" rule; the instruction "check **the** basket out" doesn't disambiguate. One reading must be the graded one; the agent picks inconsistently.
- **Failure codes:** F8 (3 runs ran out of budget chasing store-inventory paths and never submitted — note the wasted steps re-finding `/proc/locations/<City>/store-*.json` after `find` returned nothing for bare store_id), F9 (gate loops on `facts.actor` source + a `s`-typo ReferenceError), F12 (policy-interpretation split)
- **Steering opportunity:** Add a navigation-hardening line resolving the "the/my basket" + multiple-active case deterministically (the prompt already has the rule but the agent doesn't trust it under ambiguity): e.g. "checkout with an unspecified basket + ≥2 active baskets → ALWAYS OUTCOME_NONE_CLARIFICATION; never auto-pick the latest unless the user said 'latest/newest'." Also add: store records live at `/proc/locations/<City>/<store_id>.json` — map store_id→City before `find`, don't `find` bare store_id (3 runs burned 4+ steps on this and 3 timed out).

## t055 — archived-fraud TSV: sum fraudulent payment amount + cite each fraud row
- **Instruction:** "Risk Ops is reviewing a two-year-old archive export... Read /archive/payment_batch_export_<id>.tsv, identify the archived payment rows that belong to fraud incidents, and do not modify files. Answer message must contain only the total fraudulent payment amount, formatted exactly as: EUR %d.%02d ... cite every fraud row using exactly this row-ref format: /archive/...tsv#row=<RowID>"
- **Type:** investigative + structured
- **Runs analyzed:** 9
- **What the agent did:** Reads the TSV (different file/seed each run), then heuristically labels "fraud" rows by shared device/payment-method fingerprint, impossible travel (distant cities minutes apart), and tight time bursts. Sums `amount_cents`, formats `EUR x.yy`, and tries to cite each fraud row with the `#row=<RowID>` fragment. It correctly treats the task as analytical (no refusal). The dominant struggle is mechanical: `scratchpad.cite(path#row=ID)` is REJECTED ("path was not read this trial") and the agent burns 3-6 steps rediscovering that it must instead write `scratchpad.refs_why[path#row=ID]=reason` directly (which bypasses the read-check).
- **Answers across runs:** EUR 5388.10 | 8188.70 | 8766.30 | 5573.00 | 9966.80 | 10260.00 | 5722.10 — all distinct (different seed files), plus 2 no-answer (steps=1 and steps=2, never submitted).
- **Outcomes across runs:** OUTCOME_OK ×7, undefined/ERR ×2
- **Variance:** N/A on value (each seed is a different file). But the fraud-row SELECTION is wildly inconsistent in scale — e.g. one run for the same shape of file flags 19 rows, another 25, another 27→40 (run 65eb5b literally jumped 27→40 rows mid-analysis), another 24. The selection criteria are ad-hoc and self-revised.
- **Inferred verdict:** UNCERTAIN, leaning LIKELY-WRONG on most runs — there is NO ground truth visible to the agent for "which rows are fraud," and its own row-count swings (27→40 in one run; 19 vs 25 vs 40 across runs of the same archive shape) show the fraud-detection heuristic is unstable. A total that depends on a self-invented fraud threshold is very unlikely to match the grader's exact expected sum + exact fraud-row set. The 65eb5b run even saw its own prior verify expect EUR 2627.80 then overrode it to 5573.00 — strong evidence the agent has no stable notion of the correct fraud set.
- **Failure codes:** F9 (heavy — `#row=` cite gate-loop dominates every run, 3-6 wasted steps), F11 (unstable fraud-row enumeration / arithmetic over a self-chosen set), F8 (2 runs no-answer), latent F10 (the answer total ↔ cited-row set must agree; with a guessed set this is fragile).
- **Steering opportunity:** HIGH VALUE, two parts. (1) Mechanical: add an explicit nav-hardening rule — "To cite row fragments like `path#row=ID`, do NOT call `scratchpad.cite()` on the fragment (it checks read-set and rejects). Read the base file once, then write `scratchpad.refs_why['path#row=ID'] = reason` directly for each row." This alone saves ~4 steps/run and would have let the 2 no-answer runs finish. (2) Analytical: the prompt gives fraud heuristics but no decision rule for INCLUSION boundaries; give a crisp, deterministic fraud rule (e.g. "a row is fraud iff its device_fingerprint OR payment_method_fingerprint is shared across ≥2 distinct customer_ref within the file, OR it participates in an impossible-travel pair for one customer") so the fraud set stops drifting run-to-run.

## Chunk-06 rollup

**Most important cross-task patterns:**
1. **Many of my "tasks" are actually 9 different seeds of the same template** (different actor, basket ID, return status, role, file). Apparent answer "flips" (t051, t057, t058, t059) are mostly correct seed-driven divergence, NOT steering failures. The agent's per-seed logic is consistent. Real instability shows up only in t051 (policy-interpretation ambiguity) and t055 (heuristic fraud selection).
2. **SQL is effectively DOWN across the whole chunk** (MS SQL cluster timeout / silent-empty). The nav-hardening "empty ≠ absence" rule is working — every run correctly falls back to the filesystem. But this fallback costs 2-4 steps per task EVERY task, and the agent re-discovers the `/proc/...` layout from scratch each time.
3. **Two recurring gate-loops burn budget on almost every task:** (a) the `facts.actor`/`actor_roles` "non-null value but no workspace-path source" rejection (the agent keeps trying to source the `/bin/id` identity), and (b) literal-token churn (declaring `<YES>/<NO>` or both `<APPROVE>/<DENY>` then having to clear one). Combined these cost ~1-3 steps on t051/t052/t053/t054/t057/t058/t059 and contributed to the no-answer/budget failures.
4. **The `path#row=ID` citation mechanism is a trap (t055):** `scratchpad.cite()` rejects fragment paths; the agent must write `scratchpad.refs_why[fragment]=reason` directly. EVERY t055 run wasted 3-6 steps relearning this, and 2 runs never finished.
5. **Citation breadth is inconsistent and under-specified** (t054, t056): on neutral own-record lookups and "cite the OCR" tasks, some runs cite minimally and others pad refs with policy docs / every catalog file read — a latent OVERCITE risk with no clear rule.

**Dominant failure codes:** F9 (gate-loop friction) and F8 (budget no-answer) are the most pervasive across the chunk; F0 dominates the refusal/lookup/delete tasks (t052, t053, t054, t057, t058, t059, t060 are mostly well-grounded). The genuine correctness risks are concentrated in **F11/F12 on t055** (unstable fraud enumeration) and **F12/F3 on t051 + t056** (policy-interpretation ambiguity; possible un-written export file + over-citation).

**Verdict tally (across 10 tasks):** LIKELY-CORRECT: t052, t053, t054, t057, t058, t059, t060 (7). UNCERTAIN/MIXED: t051 (policy-interpretation flip), t056 (write-confirmation + citation ambiguity). LIKELY-WRONG (most runs): t055 (heuristic fraud sum with no ground truth and drifting row sets). Tasks that genuinely FLIP in a *steering-relevant* way: t051 (clarify-vs-auto-checkout) and t055 (fraud-set size swings 19↔40 within/across runs).

**Top prompt-change recommendations:**
1. **Kill the two universal gate-loops** (motivated by t051/t052/t053/t054/t057/t058/t059). Add to nav-hardening: "Identity/roles from `/bin/id` are a runtime probe — leave their facts slots `value:null, confidence:'pending'`; never try to give them a workspace-path source. Declare exactly the ONE literal token your decision implies (`<APPROVE>` XOR `<DENY>`, etc.), never both." This recovers budget on 7 of my 10 tasks and would likely save the budget-failed runs.
2. **Document the row-fragment citation idiom** (motivated by t055, the single worst task): "To cite `path#row=ID` refs, read the base file once, then set `scratchpad.refs_why['path#row=ID']=reason` directly — do NOT call `scratchpad.cite()` on a fragment (it checks the read-set and always rejects)." Pair with a deterministic fraud-inclusion rule so the fraud set stops drifting.
3. **Add a "create-file" + citation-scope rule** (motivated by t056, and t054): "When asked to create/write a report at PATH, you MUST `harness.write` it and confirm success before answering the path. Cite exactly the grounding refs the instruction names (e.g. 'cite the OCR path') — do not pad refs with every record you read while computing; on a neutral own-record lookup cite only that record."
